垃圾回收（Garbage Collection，GC）是 JVM 自动内存管理的核心。理解 GC 机制不仅能帮助你写出更高效的代码，更是 JVM 调优的必备知识。

## 如何判断对象已经"死亡"

JVM 判断对象生死主要有两种算法。

### 引用计数法（Reference Counting）

给每个对象维护一个引用计数器。有人引用它，计数器 +1；引用失效，计数器 -1。当计数器归零时，说明没人在用这个对象了。

这个算法简单高效，但有一个**致命缺陷——循环引用**：

```java
public class CircularReference {
    public Object instance;

    public static void main(String[] args) {
        CircularReference a = new CircularReference();
        CircularReference b = new CircularReference();

        a.instance = b;  // a 引用 b
        b.instance = a;  // b 引用 a

        a = null;
        b = null;
        // 此时 a 和 b 相互引用，计数器都是 1
        // 但实际上它们已经不可达，应该被回收
    }
}
```

正因为这个问题，**主流 Java 虚拟机（HotSpot）不使用引用计数法**。

### 可达性分析算法（Reachability Analysis）

这是 JVM 真正使用的算法。它的思路像遍历一棵树：

1. 从一系列被称为 **GC Roots** 的根对象开始
2. 沿着引用链向下搜索
3. 如果一个对象到 GC Roots 之间有连通的引用链，则**存活**
4. 如果从 GC Roots 出发无法到达某个对象，则**可回收**

**哪些对象可以作为 GC Roots？**

结合上一篇讲的内存布局，GC Roots 主要分布在"私有区"和"元空间"：

1. **虚拟机栈中引用的对象**（方法里的局部变量）
2. **方法区中静态属性引用的对象**（类的 `static` 变量）
3. **方法区中常量引用的对象**（字符串常量池里的引用）
4. **本地方法栈中 JNI 引用的对象**（Native 方法引用的对象）
5. **被同步锁持有的对象**

## 四种引用类型：引用的"力度"

为了让 GC 更加灵活，Java 将引用分成了四种，力度由强到弱：

| 引用类型 | 回收时机 | 典型应用场景 |
|---------|---------|-------------|
| **强引用** | 永不回收（宁可 OOM） | `Object obj = new Object()` |
| **软引用** | 内存不足时回收 | 缓存、图片加载 |
| **弱引用** | 下次 GC 时必回收 | WeakHashMap、ThreadLocal |
| **虚引用** | 随时可能回收 | 跟踪对象被回收的活动 |

```java
// 软引用示例：内存敏感的缓存
SoftReference<byte[]> cache = new SoftReference<>(new byte[1024 * 1024]);
byte[] data = cache.get(); // 内存充足时可以获取，不足时返回 null

// 弱引用示例
WeakReference<Object> weak = new WeakReference<>(new Object());
System.gc(); // GC 后 weak.get() 返回 null
```

## 垃圾收集算法

### 复制算法（Copying）：新生代的"闪电侠"

**原理**：将内存分为两块，每次只使用其中一块。GC 时将存活对象复制到另一块，然后清空当前块。

**优化（Appel 式回收）**：现代 JVM 把新生代分成 **Eden : S0 : S1 = 8 : 1 : 1**，每次只浪费 10% 的空间。

**为什么新生代适合它？**

- 新生代 98% 的对象都是"朝生夕死"
- 只需复制 2% 的存活对象，效率极高
- 不产生内存碎片

**缺点**：如果存活对象太多，复制成本会很高。

### 标记-清除算法（Mark-Sweep）：所有算法的鼻祖

**原理**：
1. **标记阶段**：遍历所有对象，标记出存活的
2. **清除阶段**：回收未被标记的对象

**优点**：速度快，不需要移动对象。

**缺点**：会产生大量不连续的内存碎片，可能导致大对象分配失败。

**应用**：CMS 垃圾收集器的核心算法。

### 标记-整理算法（Mark-Compact）：老年代的"压路机"

**原理**：
1. **标记阶段**：与标记-清除相同
2. **整理阶段**：将所有存活对象向内存一端移动，然后清理边界外的内存

**优点**：不产生碎片，内存规整。

**缺点**：最慢的算法，需要移动对象并更新所有引用指针。

**为什么老年代适合它？** 老年代对象存活率高，如果用复制算法要复制的对象太多；如果用标记-清除，碎片会积累到难以容忍。

### 分代收集理论（Generational Collection）

这是现代 JVM 采用的策略——**因地制宜**：

- **新生代**（存活率低）→ **复制算法**：快速、高效、无碎片
- **老年代**（存活率高）→ **标记-整理** 或 **标记-清除**：空间利用率高

## Minor GC：新生代的快速清扫

### 工作流程

想象你有一个很乱的房间（Eden 区），里面 98% 是垃圾，只有 2% 是宝贝。

**方案对比**：
- **标记-清除**：一个个检查、清理，累且碎片多
- **复制算法**：把 2% 的宝贝捡出来放到干净的小箱子（Survivor），然后整个房间一键清空

JVM 选择了后者。

### Eden 8:1:1 的布局是怎么干活的？

![Minor GC 流程](/images/blog/minor-gc.png)

**为什么是 8:1:1？** 研究表明新生代 98% 的对象都是短命鬼。用 80% 空间装新对象，10% 装幸存者，10% 备用。内存利用率达到 90%。

### 分配担保机制

如果某次 GC 存活对象太多，Survivor 装不下怎么办？

这时**老年代**出来收留这些对象，这叫**分配担保（Handle Promotion）**。

```
正常情况：Eden → Survivor → 老年代（熬过 15 次 GC）

担保情况：Eden → Survivor 装不下 → 直接进老年代
```

**为什么叫"过早晋升"是问题？**

原本老年代是给长命对象（Spring Bean、连接池）养老的。如果大量临时对象提前挤进来，老年代很快满了，就会触发代价高昂的 Full GC。

**优化手段**：
- `-Xmn`：增大新生代，让 Eden 和 Survivor 有更大缓冲
- `-XX:SurvivorRatio`：调整 8:1:1 比例，短命对象多就把 Survivor 调大
- `-XX:MaxTenuringThreshold`：调整晋升年龄阈值

## Full GC：整栋大楼的大扫除

Full GC 是 JVM 中开销最大的操作。它会清理**新生代 + 老年代 + 元空间**，并且会触发 **Stop The World（STW）**——暂停所有用户线程。

### GC 类型对比

| 类型 | 回收区域 | 触发频率 | 耗时 |
|-----|---------|---------|------|
| Minor GC | 新生代 | 频繁 | 毫秒级 |
| Major GC | 老年代 | 较少（仅 CMS） | 较长 |
| Full GC | 全堆 + 元空间 | 应该很少 | 秒级 |

### Full GC 触发条件

1. **老年代空间不足**：分配担保挤进来太多对象，或大对象直接分配
2. **元空间不足**：动态加载类过多（反射、CGLib、动态代理）
3. **统计预测失败**：JVM 预判下次晋升对象大于老年代剩余空间
4. **显式调用 `System.gc()`**：建议禁用（`-XX:+DisableExplicitGC`）
5. **Concurrent Mode Failure**：CMS 回收速度跟不上产生垃圾的速度

### STW 的代价

在 Full GC 期间，用户会感受到：**界面卡死、接口超时、服务无响应**。

- Minor GC 的 STW 通常只有几毫秒，感知不到
- Full GC 在大内存机器上可能卡住**几秒甚至十几秒**

### 性能优化的黄金法则

**目标：减少 Full GC 频率，甚至消除 Full GC。**

如果监控发现 Full GC 频繁（几分钟一次），通常意味着：

| 现象 | 可能原因 | 解决方案 |
|-----|---------|---------|
| Full GC 频繁 | 内存泄漏 | 排查静态集合、未关闭资源 |
| 老年代增长快 | 新生代太小 | 增大 `-Xmn` |
| 大对象直接进老年代 | 大数组/字符串滥用 | 代码优化、调整阈值 |

## 总结

| 知识点 | 核心要点 |
|-------|---------|
| 对象判活 | 可达性分析，从 GC Roots 出发 |
| 引用类型 | 强 > 软 > 弱 > 虚，灵活控制对象生命周期 |
| 收集算法 | 新生代用复制，老年代用标记-整理 |
| Minor GC | 频繁、快速、8:1:1 布局 |
| Full GC | 尽量避免，关注 STW 时间 |

掌握了这些原理，你就能更好地理解各种垃圾收集器的设计思路，以及如何针对具体场景进行 JVM 调优。
