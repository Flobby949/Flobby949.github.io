上一篇我们讲了垃圾回收的判定算法和收集策略。如果说回收算法是"招式"，那么**垃圾收集器就是真正操刀上阵的"清理工人"**。

在 Java 几十年的演进中，这些"工人"换了一代又一代。每次换代，都是对一个核心矛盾的新解法：**吞吐量 vs. 停顿时间**。

## 停顿（STW）：一切优化的出发点

在理解各种收集器之前，必须先搞清楚一个词：**Stop The World（STW）**。

GC 工作时，需要扫描对象的引用关系。如果扫描过程中用户线程还在不断创建、修改对象，引用图就会一直在变，分析结果就会不准确。于是 JVM 的解决方案是：**暂停所有用户线程，让世界静止**，等 GC 分析完再继续。

这段暂停时间，就是 STW。它是用户感受到"卡顿"的根源，也是几十年来 GC 研究者们最想缩短的东西。

## 第一代：Serial 系列——单线程工作的"独行侠"

这是最早期的收集器，设计于内存只有几十 MB 的年代。

### Serial（新生代）

- **策略**：单线程，GC 时必须 STW，使用**复制算法**。
- **特点**：简单高效，没有线程切换和协调的开销。
- **现状**：仍是客户端（Client）模式和嵌入式设备的默认选择。内存小、单核 CPU 的场景下反而很合适。

### Serial Old（老年代）

- **策略**：单线程，使用**标记-整理算法**。
- **特点**：Serial 的老年代搭档，同时也是 CMS 出现并发失败时的兜底方案。

## 第二代：Parallel 系列——多核时代的"吞吐量机器"

随着服务器 CPU 变成多核，串行 GC 显然是在浪费资源。Parallel 系列的思路很直接：**用多个线程同时做 GC**。

### Parallel Scavenge（新生代）

- **策略**：多线程，使用复制算法，GC 时仍需 STW。
- **关键词**：**吞吐量优先**。
- **核心参数**：
  - `-XX:MaxGCPauseMillis`：控制最大停顿时间（毫秒）
  - `-XX:GCTimeRatio`：控制吞吐量比例（GC 时间 / 总运行时间）
  - `-XX:+UseAdaptiveSizePolicy`：开启自适应调节策略，JVM 自动调整 Eden/Survivor 大小

```bash
# JDK 8 默认配置，适合后台计算任务
java -XX:+UseParallelGC -XX:GCTimeRatio=99 -jar app.jar
```

- **现状**：**JDK 8 的默认 GC**，适合后台批处理、数据分析等不需要频繁与用户交互的场景。

### Parallel Old（老年代）

- **策略**：多线程，使用标记-整理算法。
- **特点**：Parallel Scavenge 的老年代搭档，JDK 6 才出现，在此之前只能和 Serial Old 配对。

## 第三代：CMS——第一个追求"低停顿"的并发收集器

进入互联网时代，用户对"卡顿"极其敏感。**CMS（Concurrent Mark Sweep）** 是 JVM 史上第一个真正实现并发（用户线程和 GC 线程同时工作）的收集器。

### 四个阶段

| 阶段 | 是否 STW | 说明 |
|------|---------|------|
| 初始标记 | **是**（极短） | 只标记 GC Roots 直接关联的对象 |
| 并发标记 | 否 | 从 GC Roots 出发，顺着引用链遍历整个图，和用户线程并发进行 |
| 重新标记 | **是**（短） | 修正并发标记期间用户线程导致的变动 |
| 并发清除 | 否 | 回收死亡对象，和用户线程并发进行 |

CMS 的聪明之处在于：把耗时最长的标记和清除阶段都做成了**并发**的，STW 只发生在开始和修正的短暂时刻。

### 三个痛点

但 CMS 也有无法绕过的硬伤：

1. **内存碎片**：使用标记-清除算法，不做内存整理。随着时间推移，老年代会出现越来越多的碎片，最终连大对象都放不下。

2. **浮动垃圾（Floating Garbage）**：并发清除时，用户线程还在产生新垃圾。这些"刚死的对象"当次 GC 来不及处理，只能留到下次。因此 CMS 不能等老年代满了再触发，需要预留一部分空间（默认 92% 时触发）。

3. **并发失败（Concurrent Mode Failure）**：如果浮动垃圾产生太快，预留空间也用完了，CMS 会退化成 **Serial Old**——单线程 STW 清理整个老年代，停顿时间剧增，这是 CMS 最致命的问题。

```bash
# CMS 的典型配置
java -XX:+UseConcMarkSweepGC \
     -XX:CMSInitiatingOccupancyFraction=70 \  # 老年代 70% 时触发
     -XX:+UseCMSCompactAtFullCollection \       # Full GC 时做内存整理
     -jar app.jar
```

## 第四代：G1——划时代的 Region 架构

从 JDK 9 开始，服务器内存动辄几十 GB，传统的"全堆扫描"已经太慢。**G1（Garbage First）** 从根本上重构了堆内存的组织方式。

### 核心创新：打碎内存，按需收集

G1 把整个堆切分成大小相等的**小块（Region，通常为 1~32 MB）**，不再有物理上连续的新生代和老年代——每个 Region 可以根据需要扮演 Eden、Survivor 或 Old 角色。

同时引入了一种特殊的 Region：**Humongous**，专门存放超过 Region 大小 50% 的大对象。

```
+--------+--------+--------+--------+--------+
| Eden   | Old    | Eden   | Surv   | Humong |
+--------+--------+--------+--------+--------+
| Surv   | Old    | Free   | Eden   | Old    |
+--------+--------+--------+--------+--------+
| Old    | Free   | Eden   | Old    | Free   |
+--------+--------+--------+--------+--------+
```

### Garbage First：回收价值最高的先收

G1 会给每个 Region 维护一个**回收价值评分**（存活对象少、回收时间短的 Region 价值高）。每次 GC 时，优先选择价值最高的那批 Region 来收集——这也是"Garbage First"名字的由来。

### 可预测的停顿

G1 最核心的卖点：**你可以告诉它"每次停顿不超过 X 毫秒"，它会尽量遵守。**

```bash
# G1 典型配置
java -XX:+UseG1GC \
     -XX:MaxGCPauseMillis=200 \    # 目标停顿时间 200ms
     -XX:G1HeapRegionSize=16m \    # Region 大小
     -jar app.jar
```

- **现状**：**JDK 9 之后的默认 GC**，面试中被问到频率最高。适合大内存（4GB+）、对响应时间有要求的服务端应用。

## 第五代：ZGC——毫秒级停顿的"黑科技"

如果说 G1 是在 Region 层面做了优化，ZGC 则是从底层重写了整个 GC 的工作方式。

### 核心技术

- **染色指针（Colored Pointers）**：在 64 位指针的高位藏入元数据（对象状态、转移标记等），使 GC 可以在不扫描对象头的情况下知道对象状态。
- **读屏障（Load Barrier）**：在每次读取对象引用时插入一小段逻辑，实现并发移动对象而不需要 STW。

这两项技术让 ZGC 把几乎所有的 GC 工作都移到了**后台并发**完成，STW 阶段只剩下极短暂的初始标记和再标记。

### 停顿时间

无论堆是 8 GB 还是 16 TB，ZGC 的 STW 时间都稳定在 **1ms 以内**（JDK 16+ 的实测数据）。

```bash
# ZGC 配置
java -XX:+UseZGC \
     -Xmx16g \
     -jar app.jar
```

- **现状**：JDK 11 引入（实验性），JDK 15 生产可用，JDK 21 引入分代 ZGC 进一步提升吞吐量。适合对延迟极其敏感的场景（金融交易、实时计算）。

## 收集器的搭配组合（JDK 8 时代）

在 G1 普及之前，JDK 8 时代的收集器需要"分工搭配"——新生代和老年代各用一个：

| 新生代 | 老年代 | 特点 | 适用场景 |
|-------|-------|------|---------|
| **ParNew** | **CMS** | 低延迟（曾经的经典组合） | 响应式 Web 应用 |
| **Parallel Scavenge** | **Parallel Old** | 高吞吐（JDK 8 默认） | 后台批处理、数据计算 |
| **Serial** | **Serial Old** | 简单稳定 | 客户端、嵌入式 |

> G1 和 ZGC 不需要搭配，它们独自管理整个堆。

## 如何选择收集器

| 场景 | 推荐收集器 | 理由 |
|-----|----------|------|
| 后台批处理、大数据计算 | Parallel GC | 最大化 CPU 利用率，追求吞吐量 |
| 普通 Web 服务（JDK 8） | ParNew + CMS | 低延迟，响应用户请求 |
| 普通 Web 服务（JDK 9+） | G1 | 默认选择，平衡吞吐与停顿 |
| 大内存（>16GB）+ 低延迟 | ZGC / Shenandoah | 亚毫秒级停顿 |
| 容器、微服务（内存受限） | SerialGC | 占用少，启动快 |

## 总结

| 收集器 | 时代特征 | 核心取舍 | 默认版本 |
|-------|---------|---------|---------|
| Serial / Serial Old | 单线程时代 | 简单，STW 长 | - |
| Parallel / Parallel Old | 多核吞吐时代 | 高吞吐，STW 可接受 | JDK 8 |
| CMS | 低延迟探索 | 并发标记，但有碎片 | - |
| G1 | Region 革命 | 可预测停顿，全能型 | JDK 9+ |
| ZGC | 亚毫秒时代 | 极低停顿，黑科技实现 | JDK 15+ 生产可用 |

GC 的演进史，本质上是一部用更复杂的工程手段，不断压榨 STW 时间的历史。理解了每代收集器的设计动机，你就能在面对"为什么要用 G1 而不是 CMS"这类问题时，给出真正有深度的回答。
