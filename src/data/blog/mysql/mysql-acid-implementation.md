我们常说事务要满足 ACID 特性，但 MySQL 到底是怎么在物理层面实现这些特性的？本文将拆解 InnoDB 的日志系统，看看原子性、持久性是如何从概念落地为代码的。

## 原子性：Undo Log 的回滚魔法

**原子性**要求事务要么全部成功，要么全部失败，不存在"执行一半"的中间状态。

### 实现机制

InnoDB 通过 **Undo Log（回滚日志）** 来保证原子性。每当你执行一条修改语句，InnoDB 都会记录它的"反向操作"：

| 你的操作 | Undo Log 记录 |
|---------|--------------|
| `INSERT` 一条记录 | 对应的 `DELETE` |
| `UPDATE` 一条记录 | 更新前的旧值 |
| `DELETE` 一条记录 | 被删除的完整数据 |

### 回滚流程

当事务报错或你手动执行 `ROLLBACK` 时，MySQL 会逻辑上执行这些反向操作，把数据恢复到事务开始前的状态。

> Undo Log 不仅支撑回滚，还是 MVCC 的基石——通过版本链让并发事务能读到各自"应该看到"的历史版本。

## 持久性：Redo Log 与 WAL 技术

**持久性**要求事务一旦提交，修改就是永久的，即使数据库宕机也不能丢失。

### 面临的问题

如果每次提交都直接把数据页写回磁盘，会面临两个问题：

1. **随机 I/O**：数据页分散在磁盘各处，随机写性能极差
2. **写放大**：可能只改了一行，却要刷整个 16KB 的数据页

### WAL：先写日志，再写数据

InnoDB 采用 **WAL（Write-Ahead Logging）** 技术解决这个问题：

```
事务修改 → 写 Redo Log Buffer（内存）
         → 事务提交时，顺序写 Redo Log File（磁盘）
         → 后台线程异步刷数据页
```

**Redo Log 是物理日志**，记录的是"在某个数据页的某个偏移量做了什么修改"，而不是 SQL 语句本身。

### Crash-Safe：崩溃恢复

如果数据库在数据页还没刷盘时宕机了：

1. 重启后，MySQL 读取 Redo Log
2. 把没来得及写盘的修改"重做"一遍
3. 数据恢复到宕机前的状态

这就是 InnoDB 的 **Crash-Safe** 能力。

## 隔离性：MVCC + 锁

隔离性确保并发事务之间互不干扰。InnoDB 通过两套机制实现：

- **MVCC**：让读操作看到一致性的历史快照，实现"读不阻塞写"
- **行锁 + 间隙锁**：控制写操作的并发，防止数据冲突

> 关于 MVCC 的详细原理，参见 [深入理解 MySQL MVCC](/posts/mysql-mvcc-deep-dive)。

## 一致性：最终目标

一致性是 ACID 的终极目的，它要求数据库从一个一致状态转换到另一个一致状态。

一致性并不是靠某个单独的机制实现的，而是**原子性 + 持久性 + 隔离性**共同保障的结果，再加上业务层面的约束（如余额不能为负、外键完整性等）。

## 核心难点：二阶段提交

在 MySQL 架构中，存在两套日志系统：

| 日志 | 所属层 | 作用 |
|-----|-------|-----|
| Redo Log | InnoDB 存储引擎层 | 崩溃恢复，保证持久性 |
| Binlog | Server 层 | 归档、主从复制 |

### 为什么需要二阶段提交？

假设没有协调机制，可能出现这种情况：

- Redo Log 写成功了，Binlog 没写成功
- 主库通过 Redo Log 恢复了数据
- 从库因为没有 Binlog，数据不一致

**二阶段提交（2PC）** 就是为了解决这个问题。

### 提交流程

```
1. Prepare 阶段
   └─ InnoDB 将 Redo Log 写入磁盘，标记为 prepare 状态

2. Write Binlog
   └─ Server 层将 Binlog 写入磁盘

3. Commit 阶段
   └─ InnoDB 将 Redo Log 状态改为 commit
```

### 崩溃恢复逻辑

如果数据库在第 2 步和第 3 步之间宕机：

1. 重启后，MySQL 发现 Redo Log 处于 `prepare` 状态
2. 检查对应的 Binlog 是否完整
3. **Binlog 完整** → 继续提交事务
4. **Binlog 不完整** → 回滚事务

这样就确保了主库数据（Redo Log 驱动）和从库数据（Binlog 驱动）的绝对同步。

## 日志系统全景图

```

│                    Server 层                         │
│  ┌───────────────────────────────────────────────┐  │
│  │                   Binlog                       │  │

│  └───────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘

                    二阶段提交协调
                         │
┌─────────────────────────────────────────────────────┐
│                  InnoDB 存储引擎                     │
│  ┌─────────────────┐    ┌─────────────────────────┐ │
│  │    Undo Log     │    │       Redo Log          │ │
│  │  （回滚 + MVCC） │    │  （持久性 + Crash-Safe）│ │

└─────────────────────────────────────────────────────┘
```

## 总结

| ACID 特性 | 实现机制 | 核心作用 |
|----------|---------|---------|
| 原子性 | Undo Log | 记录反向操作，支撑回滚 |
| 持久性 | Redo Log + WAL | 顺序写日志，崩溃后重做 |
| 隔离性 | MVCC + 锁 | 读写不互斥，写写互斥 |
| 一致性 | 以上三者 + 业务约束 | 最终目标 |

二阶段提交是连接 Redo Log 和 Binlog 的桥梁，确保了单机事务和主从复制的数据一致性。理解了这套日志系统，你就掌握了 MySQL 事务的物理本质。
